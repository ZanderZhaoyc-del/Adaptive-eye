# Vision Contrast Review

Use this skill after Adaptive Eye has produced a JSON contrast report. Treat the DOM audit as the source finding list, and use the vision model only to review findings whose DOM `contrastRatio` is exactly `1`.

## Triggers

Use this workflow when any of these are true:

- The report JSON has `summary.issuesFound > 0`, but only findings with DOM `contrastRatio` equal to `1` enter vision review.
- The report JSON has `fallbackRequired: true` and a `screenshotPath`.
- The user asks for visual review, AI review, VLM fallback, false-positive review, or manual-like checking of contrast issues.
- An annotated screenshot exists and the user wants a second pass on the marked elements.

Do not use this skill as a replacement for the DOM audit. It is a narrow follow-up review step for the most suspicious `1:1` DOM contrast results.

## Inputs

Start from the JSON report generated by:

```bash
node adaptive-eye-cli/src/cli.js page <url> --report both
```

Read these fields when available:

- `pageUrl`, `pageTitle`, `generatedAt`
- `summary.issuesFound`, `summary.criticalIssues`, `summary.warningIssues`
- `findings[].text`, `tag`, `color`, `backgroundColor`, `fontSize`, `fontWeight`, `contrastRatio`, `severity`, `boundingBox`
- `cleanScreenshotPath`
- `screenshotPath`
- `annotatedScreenshotPath`
- `fallbackRequired`
- `warnings`

Screenshot priority (highest first): `cleanScreenshotPath` > `annotatedScreenshotPath` > `screenshotPath`. The bundled script applies this order automatically. A clean (overlay-free) screenshot is preferred because annotation overlay boxes can distort the model's perception of nearby pixels; the model uses each finding's `boundingBox` (CSS pixels relative to the full-page screenshot origin) to locate elements.

If no screenshot field is set, run the annotation workflow or capture a clean screenshot before requesting visual review.

## Model Configuration

Use an OpenAI-compatible vision endpoint. Never hard-code API keys in the repository or reports.

Expected local environment variables are loaded from the repository `.env` file before each vision review execution:

```bash
node .agents/skills/vision-contrast-review/scripts/load-env-and-run.js <vision-review-command>
```

Required `.env` keys:

```text
ADAPTIVE_EYE_VISION_API_KEY=<secret>
ADAPTIVE_EYE_VISION_BASE_URL=https://api.moonshot.cn/v1
ADAPTIVE_EYE_VISION_MODEL=kimi-k2.6
```

If a value is missing after loading `.env`, ask the user to configure it or report `vision_review_unavailable`. Do not paste secrets into generated reports.

If the configured model or endpoint does not accept image inputs, stop the visual review and record `vision_review_unavailable` with the API error summary. Do not infer visual confirmation without image evidence.

## Review Workflow

1. Read the JSON report and confirm it has findings or fallback evidence.
2. Select the screenshot:
   - Use `cleanScreenshotPath` first (preferred — no overlay distortion).
   - Use `annotatedScreenshotPath` second.
   - Use `screenshotPath` third.
   - If none exist and findings have `boundingBox`, run `node adaptive-eye-cli/src/cli.js page <url> --screenshot` (or `cli themes <url> --screenshot`) to produce a clean image, or `cli annotate <report-json>` for an annotated one.
   - If no screenshot can be produced, stop with `vision_review_unavailable`.
3. Run the bundled review script, which loads `.env` from the repository root and filters to DOM ratio `1` by default:
   ```bash
   node .agents/skills/vision-contrast-review/scripts/run-vision-review.js <report-json>
   ```
4. Prepare a compact finding packet containing only findings whose DOM `contrastRatio` is exactly `1`; preserve original report indices.
5. Load `references/vision-review-prompt.md` and fill in the report metadata, finding packet, and screenshot reference.
6. Send the prompt and screenshot image to the configured OpenAI-compatible model. The bundled script uses OpenAI-compatible chat completions with `image_url` parts.
7. Require structured JSON output from the model. If the response is not parseable, summarize it as a warning and mark individual findings `inconclusive`.
8. Write a sibling review report beside the original JSON:
   - `<original-base>-vision-review.json`
   - `<original-base>-vision-review.md`
9. Summarize confirmed issues, likely false positives, inconclusive items, model name, and output paths.

## Required Output Shape

The JSON review report should use this shape:

```json
{
  "auditType": "contrast-vision-review",
  "sourceReportPath": "reports/run/contrast-report-example.json",
  "pageUrl": "https://example.com",
  "generatedAt": "2026-06-11T00:00:00.000Z",
  "model": "kimi-for-coding",
  "screenshotPath": "reports/run/contrast-report-example-annotated.png",
  "status": "success",
  "summary": {
    "totalFindings": 30,
    "eligibleFindings": 1,
    "reviewedFindings": 1,
    "confirmed": 0,
    "likely": 0,
    "falsePositive": 1,
    "inconclusive": 0
  },
  "findings": [
    {
      "index": 1,
      "element": {
        "tag": "input",
        "text": "Search",
        "color": "rgb(255, 255, 255)",
        "backgroundColor": "rgb(255, 255, 255)",
        "fontSize": "14px",
        "fontWeight": "400",
        "boundingBox": { "x": 100, "y": 100, "width": 90, "height": 32 }
      },
      "domSeverity": "critical",
      "domContrastRatio": 1,
      "visionVerdict": "false-positive",
      "confidence": "medium",
      "reason": "The marked element is visually readable; inspect DOM color pairing manually.",
      "recommendedAction": "Treat as likely false positive until DOM color extraction is verified."
    }
  ],
  "warnings": []
}
```

Allowed `visionVerdict` values:

- `confirmed`: visual evidence supports the DOM finding.
- `likely`: visual evidence suggests a real issue but screenshot ambiguity remains.
- `false-positive`: visual evidence contradicts the DOM finding.
- `inconclusive`: screenshot, crop, overlay, or model limitations prevent a reliable judgment.

## Reporting Rules

- Keep DOM measurements and vision verdicts separate.
- Review only DOM `contrastRatio` equal to `1`; ignore other contrast issues in the vision review report.
- Include enough element information in each vision finding for the user to identify it: original index, tag, text, color, background color, font size, font weight, and bounding box.
- If a reviewed finding is `false-positive`, pass the vision review JSON to annotation so that finding is excluded from the annotated screenshot. The annotation overwrites any earlier first-pass overlay at the same `<base>-annotated.png` path.
- Do not downgrade a WCAG failure solely because the page "looks okay"; require a concrete visual reason.
- Do not claim full accessibility coverage. This reviews visible contrast findings only.
- Mention model/provider failures as warnings, not as audit passes.
- If a clean screenshot is used, rely on each finding's `boundingBox` for spatial localization; if only an annotated screenshot is available, account for overlay boxes possibly changing perception near the target.
- If bounding boxes are offscreen, tiny, hidden, or covered, mark the finding `inconclusive`.
